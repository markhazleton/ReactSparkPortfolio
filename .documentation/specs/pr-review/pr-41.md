## Findings

### H-01 🔴 Open

- Principle: Error Handling & Resilience (Mandatory)
- File: `src/contexts/ThemeContext.tsx:81`
- Supporting Evidence: `src/services/theme/themeCatalogService.ts:109-114`
- Issue: The PR introduces a remote Bootswatch metadata enrichment path, but the app never enables it. `loadThemeCatalog` only fetches external metadata when `attemptRemoteMetadata` is explicitly true, and the application initializes the catalog with `await loadThemeCatalog();` instead. That means the Bootswatch API is never attempted in normal runtime, so the promised enrichment path is dead code and the deployed selector will always use the local fallback catalog only.
- Why It Matters: This is a functional gap, not just an optimization miss. The PR summary and feature spec both describe external catalog enrichment as part of the delivered behavior. As written, users never receive remote descriptions/previews even when the external service is available.
- Fix: Call `loadThemeCatalog({ attemptRemoteMetadata: true })` during initialization, then retain the existing local fallback behavior when the fetch fails.
